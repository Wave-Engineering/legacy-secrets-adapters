# Dynamic Credential Shim

*A static DB password → an OpenBao-managed credential that rotates, so a leak self-expires — delivered behind the same file the unchanged app already reads.*

Temporal protection: instead of guarding a static credential that's valid forever, mint
short-lived ones that self-destruct. OpenBao's database secrets engine issues and rotates the
credential; the shim writes it where the app reads. A stolen backup, a leaked log, or a
compromised disk gets a credential that's already dead by the time the attacker tries it.
Only works when the credential is *mintable* (you control issuance) — opaque third-party API
keys can't use this pattern.

## Context — when you're here

- A legacy app reads a **static** database password from a file and won't be re-released.
- The secret must be **used** (recovered and replayed to Postgres), and it's **mintable** — a
  credential authority (OpenBao) can issue and rotate it (decision-guide §2, "mintable").
- A static password is valid *forever*, so a single leak (backup, log, stolen disk) is valid
  forever too. You want a leak to be **time-boxed**.

## Forces

- Rotation only helps if the consumer eventually **re-reads** — but the reader can't change.
- The rotating credential must therefore arrive **behind the reader's existing file**.
- A legacy **connection pool** dislikes a changing *username* — which steers us to static roles.

## Solution

OpenBao manages a **static role**: it owns and rotates the password of **one fixed Postgres user**
(`app_pg_user`) — stable username (pool-friendly), password no human chose or stored. A small
**shim** fetches the current managed credential and writes it to the reader's file; on rotation it
rewrites. The reader connects, unchanged. A *leaked copy* of the password stops working at the next
rotation — the leak self-expires.

## How it works

```
  OpenBao (static role: owns app_pg_user's password, rotates it)
        │  bao read database/static-creds/app-static
        ▼
   shim.py ──writes──▶ run/secrets.json ──reads──▶ legacy_reader.py ──connects──▶ Postgres
        ▲                                                                  (as app_pg_user)
        └── on rotation: re-fetch + rewrite the file; the reader reconnects with the new password
```

A password exfiltrated before a rotation no longer authenticates after it.

## Run the demo

Needs **Docker** (real OpenBao + Postgres), plus `bao` and `psql` on the host.

```bash
./demonstrate.py     # interactive: bring up → read → rotate & watch the leak die → tear down
```

By hand:

```bash
docker compose up -d && ./setup.sh             # real OpenBao + Postgres + a static role
./shim.py && ./legacy_reader.py                # the unchanged reader connects with the managed cred
bao write -f database/rotate-role/app-static   # rotate; a copy of the old password now fails
docker compose down -v
```

## Tradeoffs / residual exposure

- The plaintext credential still lands in a file the reader reads — **compose with
  [`cone-of-silence`](../cone-of-silence/)** to keep it RAM-only.
- Rotation **time-boxes** a leak to one rotation window; it is *not* instant revocation. A tighter
  window shrinks the exposure but increases rotation churn.
- The reader must **re-read on reconnect** (the shim rewrites the file). A reader that opens one
  connection and holds it forever won't feel a rotation until it reconnects/restarts — align
  `rotation_period` with that cadence (see `NOTES.md`).
- A single OpenBao is a **SPOF for rotation, not for use**: static credentials keep working if
  OpenBao is down (no consumer-side expiry); only rotation pauses. (`NOTES.md`.)

## Production hardening

- Deliver the credential through a **Cone** (tmpfs) so it's never at rest on disk.
- `rotate-root` so no human knows the database admin password (the demo does this).
- Tune `rotation_period` to the reader's reconnect cadence.
- **Dynamic roles** (a brand-new Postgres user per short lease) harden the leak window further but
  the changing username can break a legacy connection pool — a small extension, **left to the
  reader**. See `NOTES.md`.

## Bootstrap secret — out of scope

This demo hardcodes its bootstrap secrets — the **OpenBao dev root token** (`dev-only-root-token`)
and the **Postgres superuser password** — obviously and on purpose. How the shim would really
authenticate to OpenBao *without* a stored secret — the turtles-to-silicon problem — is an
orthogonal concern addressed by the bootstrap-secret pattern family (see
[`../../docs/decision-guide.md`](../../docs/decision-guide.md)). Conflating the two would obscure
this pattern's lesson.

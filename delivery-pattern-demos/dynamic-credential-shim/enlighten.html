Dynamic Credential Shim

managed, rotating credentials for a database app you can't change

A static password is valid forever — so a leak is valid forever too. A managed credential rotates, so a leaked copy self-expires.

The problem

A legacy app reads a static Postgres password from a file and won't be re-released. That password never changes, so the moment it leaks — a backup, a log line, a stolen disk, a screenshot — the leak is good forever. Rotating it normally means editing the app or its config. We can't. We want the credential to rotate and a leak to time-box itself, with the app none the wiser.

The idea, side by side

The credential the app reads is the same shape it always was — a username and a password in a file. Only its lifecycle changed: OpenBao mints and rotates it, a small shim writes it to the app's file, and the app connects exactly as before.

How it works

  OpenBao (static role: owns + rotates app_pg_user's password)
        │  bao read database/static-creds/app-static
        ▼
   shim ──writes──▶ secrets.json ──reads──▶ legacy app ──connects──▶ Postgres (as app_pg_user)
        ▲
        └─ on rotation: re-fetch + rewrite; the app reconnects with the new password.
           A password exfiltrated BEFORE the rotation no longer authenticates AFTER it.

The terminal walk-through in deck.html shows this happening against a real OpenBao + Postgres: connect, rotate, watch the stolen copy get rejected.

Static role vs dynamic role

OpenBao's database engine offers two shapes. This pattern centers on the static role because it's the one a legacy connection pool tolerates.

Static role (this pattern)Dynamic role (reader's exercise)
Identityone fixed usernamea new username per lease
What changesthe password, on a schedulethe whole credential expires, then is dropped
Leak windowuntil the next rotationuntil the lease TTL (can be minutes)
Connection poolfriendly — username never changeshostile — new logins need the new user

Dynamic roles shrink the leak window further, but the changing username breaks a pool that opens connections under one identity. For an app you can't change, the static role is the pragmatic win; swapping it for a dynamic role is a small, well-trodden extension — left to you.

The leak window — honest about what this buys

What it gives you

A stolen credential stops working within one rotation period instead of living forever. Tighten rotation_period and the window shrinks.

What it doesn't

It is not instant revocation, and the plaintext still lands in a file. Pair it with the Cone of Silence to keep that file in RAM, and tune rotation against how often the app reconnects.

Availability: a single OpenBao is a SPOF for rotation, not for use — a static credential keeps working if OpenBao is down; only rotation pauses. (A dynamic role with a hard expiry would instead fail closed. See NOTES.md.)

Related concepts

  • Cone of Silence — this pattern owns the credential's lifecycle; the Cone owns its delivery. Compose them: rotate the credential and keep it RAM-only.
  • broker-sidecar (planned) — the shim generalized into a sidecar with templating and an audit trail.
  • bootstrap-secret family (planned) — how the shim would authenticate to OpenBao without a stored secret. See the catalog's two axes in decision-guide.md.

Bootstrap secret — out of scope

This demo hardcodes its bootstrap secrets — the OpenBao dev token and the Postgres superuser password — obviously and on purpose. How the shim would really authenticate to OpenBao without a stored secret (the turtles-to-silicon problem) is an orthogonal concern addressed by the bootstrap-secret pattern family. Conflating it here would obscure the lesson.

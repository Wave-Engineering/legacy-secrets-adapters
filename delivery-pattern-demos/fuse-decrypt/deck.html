FUSE Decrypt

a transparent decrypt-on-read filesystem for legacy apps with arbitrary I/O

legacy-secrets-adapters · a walkthrough

The problem

A legacy app reads credentials from a plaintext file on disk — but does MORE than a single sequential read. It seeks, writes back, re-reads, and expects full POSIX semantics. A pipe or single-read shim cannot satisfy it. We need a real filesystem that decrypts on the fly — without touching the app.

Encrypt the secret to disk

First, we create the ciphertext backing store. The plaintext secret is encrypted
with AES-256-GCM; only ciphertext is stored on disk.

[you@host fuse-decrypt]$ python3 -c "import fusefs, json; ..." # (setup)
-> encrypted secrets.json to cipherstore/ (AES-256-GCM, 141 bytes)

Verify: disk holds only ciphertext

The backing file is pure ciphertext — the password is not present:

[you@host fuse-decrypt]$ grep 'S3cr3t-Pg-Pass' cipherstore/secrets.json || echo '(not found)'
(not found)

A hex dump confirms it is binary gibberish:

Mount the FUSE filesystem

The FUSE daemon mounts the cipherstore as a plaintext view. The key is held
in memory — never on disk in production (here: a demo keyfile).

[you@host fuse-decrypt]$ python3 fusefs.py cipherstore/ mnt/ demo.key &
mounting cipherstore/ -> mnt/ (AES-256-GCM, decrypt-on-read)

Read through the mount — plaintext!

The legacy reader opens the file at its expected path (inside the mount).
It sees plaintext — and exercises seek + write-back:

[you@host fuse-decrypt]$ ./legacy_reader.py --config mnt/secrets.json
[legacy-reader] opened mnt/secrets.json
[legacy-reader] read credentials for app_pg_user@db.internal/appdb
[legacy-reader] seek test: file size=109, re-read OK (109 bytes)
[legacy-reader] write-back: updated last_access field
[legacy-reader] re-read: write-back confirmed (last_access persisted)
[legacy-reader] connecting to db.internal/appdb as app_pg_user ...
[legacy-reader] (password used in memory to connect — never printed, never logged)

The reader sees plaintext through the mount. Seek + write-back work.

Disk is STILL ciphertext after a write

The reader wrote back through the mount. What happened on disk?

[you@host fuse-decrypt]$ grep 'S3cr3t-Pg-Pass' cipherstore/secrets.json || echo '(not found)'
(not found)

Write-back went through FUSE -> encrypted on disk. No plaintext leaked.

Unmount -> remount round-trip

Unmount, then remount. Data is persisted as ciphertext — the round-trip works:

[you@host fuse-decrypt]$ fusermount3 -u mnt/ && python3 fusefs.py cipherstore/ mnt/ demo.key &
mounting cipherstore/ -> mnt/ (AES-256-GCM, decrypt-on-read)
[you@host fuse-decrypt]$ cat mnt/secrets.json | python3 -c "import json,sys; d=json.load(sys.stdin); print(d['passwd'][:4]+'...')"
S3cr...

Remount confirmed — ciphertext persists, plaintext view restored.

The takeaway

Ciphertext at rest. Plaintext only through the FUSE mount — in memory, never on disk. The app does arbitrary POSIX I/O (seek, write, re-read) and never knows. The key lives in the daemon's memory; unmount = no plaintext anywhere.

The FUSE layer handles the hardest access pattern in the catalog: full POSIX semantics. Simpler patterns (Cone of Silence, fifo-stream) cover simpler readers. fuse-decrypt is the nuclear option — when nothing else fits.

Bootstrap secret — out of scope. The AES key here is a demo keyfile; how the daemon would really obtain it (TPM, OpenBao transit) is an orthogonal concern. Why & what: enlighten.html · the catalog.

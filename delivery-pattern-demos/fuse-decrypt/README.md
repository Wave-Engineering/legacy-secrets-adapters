# FUSE Decrypt

*a transparent decrypt-on-read filesystem for legacy apps with arbitrary I/O. Ciphertext on disk; plaintext only through the FUSE mount — the app does full POSIX I/O and never knows.*

## Context — when you're here

- A legacy application reads a secret from a **plaintext file** at a fixed path.
- You **cannot change the application** — it will not be re-released.
- The app does **arbitrary file I/O**: seeks, writes back, re-reads — not just a single sequential read.
- Simpler patterns (named pipe, tmpfs) can't satisfy the full POSIX contract the app expects.

## Forces

- Plaintext-at-rest is the vulnerability: backups, image layers, `grep`, a stray `git add`, a stolen disk.
- The reader's contract is frozen and demands full POSIX semantics (seek, write, re-read, truncate).
- A named pipe breaks on seek. A tmpfs Cone works but still materializes plaintext as a real file.
- Therefore the fix must intercept **at the filesystem layer** — presenting a normal file backed by ciphertext.

## Solution

Run a **FUSE (Filesystem in Userspace) daemon** that mounts the ciphertext directory as a plaintext
view. The daemon holds the AES-256-GCM key in memory and decrypts on every `read()`, encrypts on
every `write()`. The app opens files at the mount point and sees a normal filesystem. On the real
disk, only ciphertext exists.

## How it works

```
  cipherstore/secrets.json ─── AES-256-GCM ciphertext (nonce‖ct‖tag)
         │
         │  FUSE daemon (key in memory)
         │  decrypt on read() / encrypt on write()
         ▼
  mnt/secrets.json ◀── legacy app opens (sees plaintext)
                       seek(), write(), re-read() — all work
```

The app sees a normal file. The kernel routes I/O through the FUSE daemon (userspace). Every read
decrypts; every write encrypts. The plaintext never touches persistent storage.

## Tradeoffs / residual exposure

The FUSE approach is the most capable pattern in the catalog — and the most complex:

- **Complexity:** a userspace filesystem daemon is a non-trivial moving part (process management, crash recovery, systemd integration).
- **Performance:** every I/O operation crosses the kernel/userspace boundary twice (FUSE overhead). Fine for config files; unsuitable for high-throughput data.
- **Key in memory:** the daemon process holds the AES key; `root`, `ptrace`, `/proc/pid/mem` can read it. Mitigate with `no_new_privs`, dedicated user, ptrace restriction.
- **Availability:** if the daemon crashes, the mount disappears and the app's I/O fails. Mitigate with systemd `Restart=always` + health checks.
- **Environment-sensitive:** requires `/dev/fuse`, kernel FUSE module, `libfuse3` — not available in all containers, VMs, or hardened environments. Check at startup and fail loudly.

## Production hardening

- Run the daemon as a `Type=notify` systemd service with `Restart=always`; bind-mount the FUSE view into the app's mount namespace.
- Key from OpenBao Transit (decrypt-only policy) or TPM-sealed — never on disk.
- Dedicated user, `ProtectSystem=strict`, `NoNewPrivileges=yes`, `RestrictSUIDSGID=yes`.
- Health check: a sidecar periodically reads a canary file through the mount.
- **mmap caveat:** pyfuse3 does not support `mmap`; apps that `mmap` the config file need a kernel-mode solution (dm-crypt, fscrypt).

## Bootstrap secret — out of scope

This demo uses a keyfile (`demo.key`) for the AES-256 key — obviously and on purpose.
How the FUSE daemon would really obtain its key (TPM seal, OpenBao AppRole, Kubernetes
service-account token) is an orthogonal concern addressed by the bootstrap-secret pattern
family (see [`../../docs/decision-guide.md`](../../docs/decision-guide.md)). Conflating
the two would obscure this pattern's lesson.

## Related patterns

- **cone-of-silence** — simpler (tmpfs), covers sequential + seek readers, but plaintext exists as a RAM file
- **fifo-stream** — simplest (named pipe), zero-disk, but read-once sequential only
- **dynamic-credential-shim** — rotating credentials behind the same file (orthogonal: can combine)
- **tpm-sealed-bootstrap** — how the daemon obtains its key without a stored secret

FUSE Decrypt

a transparent decrypt-on-read filesystem for legacy apps with arbitrary I/O

Ciphertext on disk. A FUSE layer decrypts on read and encrypts on write — transparently. The legacy app does full POSIX I/O (seek, write, mmap) and never knows its file is actually encrypted at rest.

The problem

A legacy app reads its credentials from a plaintext file and won't be re-released. But unlike a simple sequential reader, this app does arbitrary file I/O: it seeks to specific offsets, writes fields back, re-reads, and expects standard POSIX filesystem semantics. A named pipe (FIFO) can't handle seeks. A tmpfs Cone works but still puts plaintext in RAM as a real file. We want:

  • Zero plaintext on disk — ever
  • Full POSIX I/O support (seek, write, re-read)
  • Transparent to the app — no changes
  • Encryption key never touches persistent storage

Power spectrum positioning

The catalog's delivery patterns form a complexity/capability spectrum:

simple
complex

fifo-stream

Named pipe. Zero disk. But: read-once, sequential only. Breaks on seek or re-read.

cone-of-silence

tmpfs (RAM file). Supports seeks and re-reads. But: plaintext exists as a file in RAM. Gone on unmount.

fuse-decrypt (this pattern)

Full FUSE filesystem. Supports seek, write-back, arbitrary I/O. Plaintext never stored — decrypted on-the-fly per syscall. Most complex, most capable.

Tradeoffs

Strengths

  • Full POSIX: seek, write, re-read, truncate
  • Zero plaintext on disk — ever
  • Transparent: app unchanged
  • Key in daemon memory only
  • Works with any file format

Residual exposure

  • Complexity: a userspace filesystem daemon is a non-trivial moving part
  • Performance: every I/O op crosses the kernel/userspace boundary twice (FUSE overhead)
  • Key in memory: the daemon process holds the key; ptrace / /proc/pid/mem can read it
  • Availability: if the daemon crashes, the mount disappears — app I/O fails
  • Environment-sensitive: requires /dev/fuse, kernel FUSE module, libfuse3 — not available in all containers/VMs

Production hardening

  • systemd integration: run as a Type=notify service with Restart=always; bind-mount the FUSE view into the app's mount namespace
  • Key source: OpenBao Transit (decrypt-only policy) or TPM-sealed; the key never touches persistent storage
  • Crash resilience: systemd auto-restarts the daemon; the app sees a brief I/O error then reconnects (or use a socket-activation pattern)
  • Minimal surface: the daemon runs as a dedicated user, no_new_privs, ProtectSystem=strict, only the cipherstore directory readable
  • mmap: pyfuse3 does not natively support mmap; apps that mmap the config file need a kernel-mode solution (dm-crypt, fscrypt) instead

Related patterns

  • cone-of-silence — simpler (tmpfs), covers sequential + seek readers, but the plaintext exists as a RAM file
  • fifo-stream — simplest (named pipe), zero-disk, but read-once sequential only
  • dynamic-credential-shim — rotating credentials behind the same file (orthogonal: can combine with fuse-decrypt)
  • tpm-sealed-bootstrap — how the FUSE daemon obtains its key without a stored secret (the bootstrap problem)

Bootstrap secret — out of scope. This demo uses a keyfile (demo.key) for the AES-256 key. How the daemon would really authenticate to its key source (TPM seal, OpenBao AppRole, Kubernetes service-account) is an orthogonal concern addressed by the bootstrap-secret pattern family.

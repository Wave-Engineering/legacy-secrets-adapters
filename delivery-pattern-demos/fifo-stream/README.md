# FIFO Stream

*named-pipe zero-disk secret delivery — the secret streams through kernel memory and never exists on any filesystem.*

The strongest spatial guarantee in the catalog: no file on any filesystem — not even a RAM-backed
one. A writer pushes the secret into a named pipe (FIFO); the app's existing `open(path)` →
`read()` receives it through kernel memory and the data vanishes. If the app reads its config
once and never seeks back, this is the simplest, most auditable answer. Breaks the moment an
app seeks, re-reads, or does mmap — for those, reach for cone-of-silence or fuse-decrypt.

## Context — when you're here

- A legacy application reads a secret from a **file at a fixed path**.
- You **cannot change the application** — it will not be re-released.
- The reader opens the file **once, reads sequentially, and never seeks back** (read-once).
- You want **zero disk footprint** — not even a tmpfs file that exists in RAM.

## Forces

- Even a RAM-backed (tmpfs) plaintext file *exists* — readable by root, other processes in the
  mount namespace, potentially paged to swap. For a read-once reader, we can do better.
- The reader's contract is frozen: *"open a file at path P, read it."*
- The fix must be **transparent to the reader** — same path, same syscall.

## Solution

Replace the file at the reader's expected path with a **POSIX named pipe (FIFO)**. A writer
feeds the secret into the pipe; the reader opens the same path and receives it. The data
streams through the kernel pipe buffer — no bytes are stored on any filesystem, not even
tmpfs.

## How it works

```
  secret source ──write──▶  mkfifo /path/secrets.json  ◀──open()── legacy app
  (OpenBao / env)           (kernel pipe buffer)                    (unchanged)
                            NO DATA STORED ANYWHERE
```

Same syscall, same path the app always used. The "file" is now a rendezvous point: data
flows through once and is gone.

## Tradeoffs / residual exposure

- **Read-once.** The pipe can only be read once. If the app re-reads the config (e.g., on HUP
  or reconnect), the second read blocks forever. This pattern is strictly for sequential,
  read-once readers.
- **Timing coordination.** The writer must feed the pipe before or concurrently with the
  reader's `open()`. Requires orchestration (systemd ordering, wrapper script, concurrent
  writer thread).
- **No persistence.** A service restart means a new write is needed — the pipe holds nothing
  between runs. The writer must be available at every start, not just first boot.
- **Process-memory exposure.** Once the reader has the secret in its process memory, the usual
  process-level attacks apply (`ptrace`, `/proc/pid/mem`, core dumps). The pipe eliminates the
  *at-rest* and *filesystem* exposure surfaces, not the in-process ones.

## Production hardening

- The pipe lives in a `RuntimeDirectory` (auto-cleaned on stop), mode `0600`, owned by the
  service user.
- The writer is an `ExecStartPre=` script that fetches from OpenBao and feeds the pipe.
  Systemd ordering guarantees the pipe is fed before the app starts.
- For apps that re-read: combine with a **broker process** that re-feeds the pipe on demand
  (watches for the reader's open via inotify on the directory).

## Bootstrap secret — out of scope

This demo hardcodes its secret (`S3cr3t-Pg-Pass`) obviously and on purpose. How the writer
obtains the real secret (authenticating to OpenBao, unwrapping a response-wrapped token,
machine identity) is an orthogonal concern addressed by the bootstrap-secret pattern family.
See [`../../docs/decision-guide.md`](../../docs/decision-guide.md).

## Related patterns

- **cone-of-silence** — tmpfs zone for readers that re-read or need persistence (weaker isolation, broader compatibility)
- **broker-sidecar** — materialize from OpenBao for an unchanged file-reader (handles re-reads)
- **fuse-decrypt** — FUSE filesystem decrypting on read (handles reads + writes)

FIFO Stream

named-pipe zero-disk secret delivery

legacy-secrets-adapters · a walkthrough

The problem

A legacy app reads its credentials from a file at a fixed path. Even a RAM-backed tmpfs file can be read by root or another process sharing the mount. For a read-once sequential reader, we can do better: deliver the secret through a named pipe (FIFO) so it streams through kernel memory and NEVER exists on any filesystem — not even tmpfs.

Create the named pipe

Replace the file the reader expects with a FIFO at the same path.
A FIFO looks like a file but is a kernel pipe buffer — no data stored anywhere.

[you@host fifo-stream]$ mkfifo /tmp/fifo-stream-demo/secrets.json
[you@host fifo-stream]$ ls -la /tmp/fifo-stream-demo/secrets.json
prw------- 1 you you 0 Jan  1 09:00 /tmp/fifo-stream-demo/secrets.json
Note the 'p' at the start — that's a pipe, not a regular file.


Writer feeds, reader reads
The writer pushes the secret into the pipe. The reader opens the same path
and receives the secret — streaming through kernel memory, never touching disk.
[you@host fifo-stream]$ echo '{"passwd": "S3cr3t-Pg-Pass", ...}' > /tmp/fifo-stream-demo/secrets.json &
[1] 12345
[you@host fifo-stream]$ CONFIG_PATH=/tmp/fifo-stream-demo/secrets.json ./legacy_reader.py
[legacy-reader] opened /tmp/fifo-stream-demo/secrets.json
[legacy-reader] connecting to db.internal/appdb as app_pg_user ...
[legacy-reader] (password used in memory to connect — never printed, never logged)
The reader gets the secret — delivered through the pipe, never stored.


Grep the disk — nothing
The secret streamed through kernel memory. There is no file to grep,
no data on disk, no tmpfs remnant — nothing.
[you@host fifo-stream]$ grep -r 'S3cr3t-Pg-Pass' /tmp/fifo-stream-demo/ --exclude='*.py' || echo '    (nothing found)'
    (nothing found)
A FIFO has no content — it's a rendezvous point, not a container. The secret
existed only in the kernel pipe buffer while data was in flight.
Zero disk footprint. Not even a tmpfs file to protect.


Second cat hangs (read-once)
A named pipe is read-once by nature. Once the data flows through, there is
nothing left to read — a second reader blocks forever (until a new writer appears).
[you@host fifo-stream]$ cat /tmp/fifo-stream-demo/secrets.json   # hangs — no writer, nothing buffered
^C  (interrupted after 5s — the cat hung, proving read-once)
Read-once: the secret flows through once and is gone. No replay possible.


The takeaway
A named pipe (FIFO) delivers a secret with zero disk footprint — the data streams through kernel memory and never exists on any filesystem. The legacy app opens the same path it always did; it has no idea the 'file' is a pipe.
Tradeoffs: read-once (the app can't re-read the file), timing coordination (the writer must feed before or concurrently with the reader), and no persistence (a restart needs a new write). Stronger than tmpfs for sequential, read-once readers.
Bootstrap secret — out of scope. How the writer obtains the secret to feed into the pipe is an orthogonal concern (the bootstrap-secret pattern family). Why & what: enlighten.html · the catalog.

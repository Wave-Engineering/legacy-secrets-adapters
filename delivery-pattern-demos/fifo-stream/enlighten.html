The problem

A legacy application reads its credentials from a file at a fixed path. Even with the Cone of Silence (tmpfs), the plaintext still exists — as a file in RAM, readable by root, by processes in the same mount namespace, potentially paged to swap. For a reader that opens the file once, reads sequentially, and never seeks back — we can eliminate the file entirely.

The idea, side by side

Replace the file with a named pipe (FIFO) at the same path. The app's open() syscall works identically — it just blocks until a writer feeds data into the pipe.

Why stronger than tmpfs

tmpfs file (Cone of Silence)

exists as a filereadable by root The plaintext exists in RAM as a file — visible to anything that can read the mount. Must be wiped explicitly. Can be paged to swap.

Named pipe (FIFO Stream)

no stored dataread-once No data is stored anywhere — the secret flows through the kernel pipe buffer and is gone. Nothing to steal, nothing to wipe, nothing to page.

The tradeoff: a tmpfs file can be read multiple times, seeked, and persists for the service lifetime. A pipe is strictly read-once and requires timing coordination between writer and reader.

Tradeoffs

Read-once

The pipe can only be read once. If the app re-reads the config file (e.g., on HUP or reconnect), the second read blocks until a new write. This pattern only works for sequential, read-once readers — apps that open, read, close, and never look back.

Timing coordination

The writer must feed the pipe before (or concurrently with) the reader's open(). If the reader opens first, it blocks. If the writer writes with no reader, it blocks. Orchestration (systemd ordering, a wrapper script, or a concurrent writer thread) is required.

No persistence

A service restart means a new write is needed — the pipe holds nothing between runs. The writer must be available at every start, not just first boot. For services that restart frequently, this adds latency (waiting for the writer) and a hard dependency on the secret source's availability.

In production

The pipe lives at the path the app already reads. The writer is typically a systemd ExecStartPre= script or a sidecar that fetches the secret from OpenBao and feeds the pipe just before the app's ExecStart=. Ordering guarantees in the service unit handle the timing coordination.

[Service]
ExecStartPre=/usr/local/bin/feed-secret --pipe /run/app/secrets.json
ExecStart=/opt/legacy-app/bin/start
RuntimeDirectory=app
RuntimeDirectoryMode=0700

The pipe is created in a RuntimeDirectory (auto-cleaned on stop), mode 0600, owned by the service user. The writer runs as the same user or via a privileged helper that drops the fd.

This pattern sits on the decision tree at: Can you change the reader? NO (transparent) → Reader's access pattern? Sequential, read-once → fifo-stream.

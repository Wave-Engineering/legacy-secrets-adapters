# Broker Sidecar

*A generalized vault-materialization sidecar: fetches any secret from OpenBao/Vault, renders it
through a Jinja2 template into whatever format the app reads, and watches for rotation — so
the app never changes, and a rotated credential is delivered automatically.*

The general-purpose pattern: any secret type, any output format, automatic rotation handling.
A long-running sidecar authenticates to the vault, fetches secrets, renders them through a
template into whatever the app expects (INI, YAML, JSON, .env), and polls for changes. When
you have dozens of apps with dozens of formats, this is the pattern that scales — at the cost
of a daemon to operate and a vault dependency to maintain. For a single DB credential with a
known format, the dynamic-credential-shim is simpler.

## Context — when you're here

- A legacy app reads credentials from a **config file in its own format** (INI, YAML, .env, etc.)
  and won't be re-released.
- The secret must come from a vault (OpenBao, HashiCorp Vault) — for rotation, auditability,
  or short-lived access.
- You **can** wrap the app's launch (add a sidecar / modify the init sequence), even though you
  can't change the app's code. Decision-guide: "Can you change the reader?" → YES (sidecar).

## Forces

- The app reads a specific config **format** — the vault's native API shape won't do.
- Rotation must be **transparent** — the app just re-reads the same file path.
- The pattern must work for **any** secret shape, not just database credentials.

## Solution

A **broker sidecar** runs alongside the app. It authenticates to OpenBao, fetches the secret
from KV v2, renders it through a Jinja2 template into the file the app reads, and polls for
new versions. On rotation (a new secret version), it re-renders. The app reconnects with the
fresh credential, unchanged.

## How it works

```
  OpenBao (KV v2: secret/apps/legacy-db)
        │  GET /v1/secret/data/apps/legacy-db
        ▼
   broker.py ──Jinja2──▶ run/db.conf ──reads──▶ legacy_reader.py
        ▲                  (INI format)          (unchanged app)
        └── polls: on new version → re-render; the app sees fresh credentials
```

This generalizes the [dynamic-credential-shim](../dynamic-credential-shim/): instead of one
hardcoded JSON output from one engine, the broker renders **any** secret through **any** template.

## Run the demo

Needs **Docker** (real OpenBao dev-mode) plus Python with `jinja2`.

```bash
./demo.py            # end-to-end: bring up → seed → fetch → render → rotate → verify → tear down
./demonstrate.py     # interactive: YOU drive each step
```

By hand:

```bash
docker compose up -d                     # OpenBao dev-mode
# seed a secret:
python3 -c "import demo; demo.seed_secret(demo.INITIAL_SECRET)"
./broker.py                              # fetch + render
./legacy_reader.py                       # the app reads the rendered config
# rotate:
python3 -c "import demo; demo.seed_secret(demo.ROTATED_SECRET)"
./broker.py                              # re-renders with the new version
./legacy_reader.py                       # app sees the new credential
docker compose down -v
```

## Tradeoffs / residual exposure

- **Operational dependency:** the broker must reach the vault to fetch/rotate. If the vault is
  unreachable, existing rendered files remain usable but cannot be refreshed.
- **Plaintext on disk:** the rendered config file contains the secret in cleartext — **compose
  with [`cone-of-silence`](../cone-of-silence/)** to keep it RAM-only.
- **Bootstrap out of scope:** how the broker authenticates to the vault is a separate concern
  (see below).
- **Poll lag:** rotation is detected on the next poll cycle, not instantly. Tune
  `BROKER_POLL_INTERVAL` against the rotation cadence.

## Production hardening

- Deliver the rendered file into a **Cone** (tmpfs) so it's never at rest on persistent disk.
- Use **AppRole** or **Kubernetes auth** instead of a static token for the broker's own
  authentication to the vault.
- Add **file-change notification** (inotify / fsnotify) downstream so the app picks up rotation
  faster than "on next reconnect".
- Run the broker under a **service manager** (systemd, container sidecar) with restart-on-failure.
- Consider **response-wrapping** for an additional layer of one-time-use security on fetch.

## Bootstrap secret — out of scope

This demo hardcodes its bootstrap secret — the **OpenBao dev root token** (`dev-only-root-token`)
— obviously and on purpose. How the broker would really authenticate to OpenBao *without* a
stored secret — the turtles-to-silicon problem — is an orthogonal concern addressed by the
bootstrap-secret pattern family (see
[`../../docs/decision-guide.md`](../../docs/decision-guide.md)). Conflating the two would obscure
this pattern's lesson.

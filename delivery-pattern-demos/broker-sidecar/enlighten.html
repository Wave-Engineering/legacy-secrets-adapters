Broker Sidecar

generalized vault-materialization: any secret, any template, any app

A sidecar process authenticates to the vault, fetches any secret, renders it through a template into the file the app reads, and watches for rotation. The app never changes — only what’s behind its config file.

The problem

A legacy app reads credentials from a config file in its own format. The credential must come from a vault (for rotation, auditability, short-lived access), rendered into that format, and refreshed when the secret rotates — all without touching the app’s code.

The dynamic-credential-shim solved a special case: one secret type (database static-creds), one output (JSON). But real estates have dozens of apps reading dozens of formats. We need the general pattern.

The shim is a special case of the broker — a broker whose template is hardcoded and whose source is a single engine. The sidecar trades that simplicity for universality.

Tradeoffs

Operational dependency

The broker must reach the vault to fetch secrets. If the vault is unreachable, the sidecar cannot render fresh credentials. (Existing rendered files remain usable until rotation.)

Plaintext on disk

The rendered config file contains the secret in cleartext. Pair with the Cone of Silence to keep it RAM-only.

Bootstrap out of scope

How the broker itself authenticates to the vault (the bootstrap secret) is a separate concern — see the bootstrap-pattern family in the decision guide.

Related concepts

  • Dynamic Credential Shim — the special case this pattern generalizes (database-only, JSON-only, one-shot).
  • Cone of Silence — deliver the rendered file into RAM so plaintext never touches persistent disk.
  • bootstrap-secret family (planned) — how the broker authenticates to the vault without a stored secret.

Bootstrap secret — out of scope

This demo hardcodes its bootstrap secret — the OpenBao dev root token (dev-only-root-token) — obviously and on purpose. How the broker would really authenticate to OpenBao without a stored secret (AppRole, Kubernetes auth, cloud IAM) is an orthogonal concern addressed by the bootstrap-secret pattern family. Conflating it here would obscure this pattern’s lesson.

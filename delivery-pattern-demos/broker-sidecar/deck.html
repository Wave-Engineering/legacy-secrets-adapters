Broker Sidecar

a generalized vault-materialization sidecar with templating and rotation watch

legacy-secrets-adapters · a walkthrough

The problem

A legacy app reads credentials from a config file in its own format. The secret must come from a vault, rendered into THAT format, and refreshed when it rotates — all without touching the app. The dynamic-credential-shim solved this for one shape (a JSON secrets file + a Postgres static role). The BROKER generalizes: any secret, any template, any format.

How this generalizes the shim

The dynamic-credential-shim is a PURPOSE-BUILT fetcher: one secret type (database/static-creds), one output format (JSON), one lifecycle (manual re-run). The broker sidecar is the GENERALIZED form:

1. Fetch ANY secret from KV v2 (not just database credentials)
2. Render through a Jinja2 TEMPLATE (the app's native config format)
3. POLL for version changes and re-render automatically
4. Log lifecycle events (fetch, render, rotation)

Same principle — a sidecar materializes what the app reads — but now it works for any secret shape the vault can store.

Bring up OpenBao + seed the secret

Real OpenBao (dev mode) in a container. The root token is an obvious dev value —
a bootstrap secret, out of scope (see the README).

[you@host broker-sidecar]$ docker compose up -d
 ✔ Container broker-sidecar-openbao-1  Started

Seed a secret into KV v2 at secret/apps/legacy-db:

[you@host broker-sidecar]$ python3 -c "import demo; demo.seed_secret(demo.INITIAL_SECRET)"
The secret is now in OpenBao: host, port, dbname, username, password (S3cr3t-Pg-Pass).


Broker fetches and renders
The broker fetches the secret and renders it through templates/db.conf.j2
into the file the legacy app reads (run/db.conf):
[you@host broker-sidecar]$ ./broker.py
[broker] fetched secret v1, rendered db.conf.j2 -> run/db.conf
The rendered output is a standard INI config — the app's native format:
[you@host broker-sidecar]$ cat run/db.conf
# Database configuration — rendered by the broker sidecar.
# The legacy app reads this file; the broker keeps it fresh.
[database]
host = 127.0.0.1
port = 5432
dbname = appdb
username = app_pg_user
password = S3cr3t-Pg-Pass


The legacy reader connects — unchanged
The legacy reader opens its config file and reads the credential.
It has no idea a broker sidecar put it there:
[you@host broker-sidecar]$ ./legacy_reader.py
[legacy-reader] opened db.conf; found credential for app_pg_user@127.0.0.1:5432/appdb
[legacy-reader] password value: S3cr3t-Pg-Pass
[legacy-reader] (in production this would be used to connect — never printed)
The reader got a vault-managed credential without any code change.


Rotate — broker detects and re-renders
Simulate rotation: write a new version of the secret to KV v2
(in production, an operator or automation writes the new value):
[you@host broker-sidecar]$ python3 -c "import demo; demo.seed_secret(demo.ROTATED_SECRET)"
The broker fetches again — it sees v2, re-renders the template:
[you@host broker-sidecar]$ ./broker.py
[broker] fetched secret v2, rendered db.conf.j2 -> run/db.conf
The reader (unchanged) sees the new credential:
[you@host broker-sidecar]$ ./legacy_reader.py
[legacy-reader] opened db.conf; found credential for app_pg_user@127.0.0.1:5432/appdb
[legacy-reader] password value: R0tated-Pg-Pass-v2
[legacy-reader] (in production this would be used to connect — never printed)
Rotation detected, re-rendered, reader sees the new value — no code touched.


The takeaway
The broker sidecar is the GENERALIZED delivery pattern: authenticate to a vault, fetch any secret, render it through a template into whatever format the app reads, and watch for rotation. The app never changes — only what's behind its config file.
The dynamic-credential-shim is a SPECIAL CASE of this pattern: a broker whose template is hardcoded as JSON and whose secret source is the database engine. The sidecar trades that specificity for universality.
Bootstrap secret — out of scope. The OpenBao token here is an obvious dev value; how the broker would really authenticate is an orthogonal concern (the bootstrap-secret pattern family). Why & what: enlighten.html · the catalog.

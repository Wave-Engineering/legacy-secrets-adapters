# TPM-Sealed Bootstrap

*Seal the materializer's bootstrap credential to the machine's TPM2 PCR state — the regress ends at silicon.*

The strongest hardware-bound bootstrap: the credential is sealed to the machine's TPM2 Platform
Configuration Registers — only this specific hardware in this specific firmware/kernel state can
unseal it. A stolen disk, a cloned image, or a tampered boot chain all produce a dead credential.
The turtles-to-silicon regress ends here. Requires TPM2 hardware (physical or vTPM) and
systemd >= 250; no cross-machine portability, and a firmware update requires a deliberate reseal.

## Context — when you're here

- A materializer (shim, sidecar, agent) needs a credential to authenticate to its key source
  (e.g. an OpenBao token), and you need that credential to **not** be a stored secret.
- The machine has a **TPM2** (physical or, for dev/test, swtpm).
- You're running **systemd >= 250** (which provides `systemd-creds` and `LoadCredentialEncrypted=`).

## Forces

- The bootstrap credential must not be a plaintext file on disk — that recreates the problem.
- It must not require human intervention at every service start (no passphrase prompts).
- It must be **bound to the machine** — stolen and replayed elsewhere, it's dead.
- A firmware/kernel update must not silently break boot without a deliberate reseal.

## Solution

Seal the bootstrap credential to the TPM2's **PCR state** via `systemd-creds encrypt`. The sealed
blob lives on disk as opaque ciphertext — useless on any other machine or after any boot measurement
change. At service start, systemd's `LoadCredentialEncrypted=` unseals it into
`$CREDENTIALS_DIRECTORY` (a per-service tmpfs, mode 0400). The materializer reads from there,
authenticates to OpenBao, and the plaintext never touches persistent storage.

## How it works

```
  Deploy time (seal ceremony):
    plaintext token --> systemd-creds encrypt --with-key=tpm2 --> sealed-blob.cred (on disk)

  Service start (automatic):
    systemd LoadCredentialEncrypted=bao-token:sealed-blob.cred
      -> TPM2 unseal (PCR policy check)
      -> plaintext in $CREDENTIALS_DIRECTORY/bao-token (tmpfs, 0400)

  Runtime:
    materializer.py reads $CREDENTIALS_DIRECTORY/bao-token --> authenticates to OpenBao --> done
```

A stolen blob is dead: wrong machine, wrong PCR state — unseal fails.

## Run the demo

Needs **systemd >= 250**, **swtpm**, and **Docker** (for OpenBao).

```bash
./demo.py            # non-interactive: seal -> opaque -> unseal -> authenticate -> tamper-test
./demonstrate.py     # interactive TUI: step through each phase
```

If systemd or swtpm is absent, both scripts exit gracefully with a pointer to `deck.html`.

## Tradeoffs / residual exposure

- **Hardware-bound.** The credential cannot migrate to another machine without re-sealing. This is
  the security property (stolen blob is dead) but also the operational cost (hardware swap requires
  a reseal ceremony).
- **PCR brittleness.** Any change to sealed measurements (firmware update, kernel upgrade, bootloader
  change) breaks unseal. The operational answer is a **reseal ceremony** after known-good updates.
- **systemd coupling.** Requires systemd >= 250 with `systemd-creds` and `LoadCredentialEncrypted=`.
  Not portable to containers without TPM passthrough, or to non-systemd inits.
- **Still trusts the key source.** The TPM authenticates the materializer to OpenBao, but you still
  trust OpenBao to issue valid credentials. The regress ends at silicon, but trust extends to the
  vault.

## Production hardening

- Bind to **specific PCRs** (`--tpm2-pcrs=7+11+14`) — firmware, kernel, shim — so the credential
  is irrecoverable if any of those change without a deliberate reseal.
- **Automate the reseal ceremony**: after a validated OS update, re-seal to new measurements in the
  same maintenance window.
- Pair with **Cone of Silence** for the delivery side — the materializer's token is bootstrap
  (this pattern); the app's secret is delivery (Cone).
- Use **systemd sandboxing** (`ProtectSystem=strict`, `PrivateTmp=yes`, `NoNewPrivileges=yes`) on
  the materializer's unit for defense in depth around the plaintext window.

## Delivery pattern — out of scope

This demo anchors the bootstrap secret (the materializer's OpenBao token sealed to the TPM). What
*consumes* the anchored credential — a delivery pattern that materializes the secret to an unchanged
app — is an orthogonal concern addressed by the delivery-pattern family (see
[`../../docs/decision-guide.md`](../../docs/decision-guide.md)). Conflating the two would obscure
this pattern's lesson.

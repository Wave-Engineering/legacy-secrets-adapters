TPM-Sealed Bootstrap

seal the materializer's credential to silicon — the regress ends at the hardware

legacy-secrets-adapters · a walkthrough

The bootstrap problem

A materializer needs a credential to authenticate to its key source (e.g. OpenBao). But where does THAT credential live? If it's a file, we're back to the same problem. If it's another vault, who authenticates to THAT vault? Turtles all the way down.

The TPM ends the regress: seal the bootstrap credential to the machine's hardware root of trust. Only THIS machine, in THIS boot state, can unseal it. The secret never exists in cleartext on disk — it goes from silicon to RAM to the wire.

Start the software TPM

For the demo we use swtpm (a software TPM2 emulator). In production this is the
physical TPM2 chip on the motherboard — /dev/tpmrm0.

[you@host tpm-sealed-bootstrap]$ swtpm socket --tpmstate dir=swtpm-state --tpm2 --ctrl type=unixio,path=swtpm-state/swtpm-sock --flags not-need-init,startup-clear &
[swtpm started, PID 12345]

Seal the credential to the TPM

Seal our bootstrap secret (here: an obvious fake) to the TPM's current PCR state.
The output is an opaque blob — ciphertext bound to this machine's measurements.

[you@host tpm-sealed-bootstrap]$ echo -n "S3cr3t-Pg-Pass" | systemd-creds encrypt --with-key=tpm2 --name=bao-token - run/bao-token.cred
[you@host tpm-sealed-bootstrap]$ ls -la run/bao-token.cred
-rw------- 1 you you 456 Jan  1 09:00 run/bao-token.cred
Is the plaintext in the blob? It must not be:
[you@host tpm-sealed-bootstrap]$ grep -c 'S3cr3t-Pg-Pass' run/bao-token.cred || echo 'not found (good)'
not found (good)
Sealed. The credential is ciphertext on disk, bound to silicon.


Unseal via the TPM
systemd-creds decrypt reverses the seal — but ONLY if the TPM's PCR state matches.
This is what systemd does at service start with LoadCredentialEncrypted=.
[you@host tpm-sealed-bootstrap]$ systemd-creds decrypt --name=bao-token run/bao-token.cred -
S3cr3t-Pg-Pass
Unsealed. In production this lands in $CREDENTIALS_DIRECTORY (tmpfs, 0400).


Materializer authenticates
The materializer reads the unsealed credential from $CREDENTIALS_DIRECTORY and
authenticates to OpenBao. The credential went: TPM -> RAM -> network. Never disk.
[you@host tpm-sealed-bootstrap]$ CREDENTIALS_DIRECTORY=/run/credentials/tpm-demo BAO_ADDR=http://127.0.0.1:58300 python3 materializer.py
[materializer] read credential 'bao-token' from $CREDENTIALS_DIRECTORY (14 chars, never printed)
[materializer] authenticated to OpenBao as 'token'
[materializer] bootstrap complete — credential never touched disk
Bootstrap complete. The materializer is authenticated without a stored secret.


Tamper test — PCR-extend breaks unseal
The TPM binding means: change the machine's boot measurements and the credential
becomes irrecoverable. Simulate by restarting swtpm with fresh state:
[you@host tpm-sealed-bootstrap]$ # restart swtpm with new state (simulates different PCR values)
[swtpm restarted with fresh measurements]
[you@host tpm-sealed-bootstrap]$ systemd-creds decrypt --name=bao-token run/bao-token.cred -
Failed to decrypt credential: TPM2 policy authorization failed
Unseal FAILED. A tampered machine cannot recover the bootstrap credential.


The takeaway
The bootstrap regress ends at silicon. The materializer's credential is sealed to the TPM — it never exists in cleartext on disk, it can only be recovered by THIS machine in THIS boot state, and a tamper (firmware change, different machine, boot attack) makes it irrecoverable.
In production: systemd's LoadCredentialEncrypted= unseals at service start into a per-service tmpfs ($CREDENTIALS_DIRECTORY, mode 0400). The materializer reads it, authenticates, and the credential lives only in RAM for the duration of the call.
PCR brittleness is real — a kernel update changes measurements and breaks unseal. The reseal ceremony (orchestrated re-sealing after a known-good update) is the operational cost. Pair with Cone of Silence for the delivery side.
Delivery pattern -- out of scope. This demo anchors the bootstrap secret (the materializer's token). What consumes the anchored credential -- a delivery pattern that materializes the secret to an unchanged app -- is an orthogonal concern (see the delivery-pattern family).

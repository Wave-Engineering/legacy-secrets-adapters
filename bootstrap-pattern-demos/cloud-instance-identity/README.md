# Cloud Instance Identity

*The machine already has an identity — use the cloud provider's instance metadata as the bootstrap credential, so no secret needs to be stored.*

## Context — when you're here

- A materializer/shim needs to authenticate to OpenBao/Vault to fetch a secret for an unchanged app.
- The workload runs on a cloud instance with a provider-assigned identity (AWS IAM instance profile,
  GCP service account, Azure managed identity).
- You refuse to store a bootstrap secret on disk — because storing it just moves the problem.

## Forces

- The turtles-to-silicon regress: every bootstrap credential has to come from *somewhere*.
- A stored token is a liability — if the disk leaks, the bootstrap credential leaks.
- Cloud instances already have a provider-assigned identity (IAM role) — it's just not being used.
- The metadata endpoint (169.254.169.254) is a hypervisor-backed service, not a network hop.

## Solution

The materializer calls the **AWS IMDSv2** metadata endpoint (a two-step PUT-then-GET flow that
prevents SSRF) to obtain short-lived AWS credentials, then authenticates to OpenBao via the
**AWS auth method**. OpenBao verifies the identity via STS GetCallerIdentity and issues a scoped
Vault token. No stored secret — the instance's IAM role IS the bootstrap credential.

## How it works

```
  EC2 Instance                    169.254.169.254 (hypervisor metadata)
       │                                 │
       │─── PUT /latest/api/token ──────>│  (TTL header; prevents SSRF)
       │<── session-token ───────────────│
       │                                 │
       │─── GET /meta-data/iam/... ─────>│  (with session token header)
       │<── {AccessKeyId, Secret, Token} │
       │                                 │
  EC2 Instance                    OpenBao / Vault
       │                                 │
       │─── POST /v1/auth/aws/login ────>│  (signed STS request)
       │<── {client_token, policies} ────│
       │                                 │
       │─── GET /v1/secret/data/... ────>│
       │<── {password: "S3cr3t-Pg-Pass"} │
```

Nothing on disk. The identity comes from the hypervisor; the auth comes from STS verification.

## Run the demo

Needs **Docker** (OpenBao dev-mode). The mock metadata service runs in-process (stdlib http.server).

```bash
./demonstrate.py     # interactive: bring up → IMDSv2 flow → materialize → rotation → tear down
```

By hand (or for CI):

```bash
docker compose up -d                  # OpenBao dev-mode
python3 demo.py                       # full end-to-end proof (exit 0 = pass)
docker compose down -v
```

## Tradeoffs / residual exposure

- **Cloud-only.** This pattern requires a cloud provider with an instance metadata service. It does
  not work on bare-metal or on-prem without a hypervisor-backed identity.
- **Trusts the provider.** The cloud control plane is the root of trust. If the provider is
  compromised (or coerced), the bootstrap credential is compromised. This is the same trust already
  placed in the provider for compute, network, and storage.
- **Metadata service exposure.** IMDSv1 was vulnerable to SSRF. IMDSv2 mitigates this (PUT requires
  a hop-limit header), but misconfiguration can still expose it. Enforce IMDSv2-only and restrict
  the IAM role to minimum permissions.
- **Machine-level, not process-level.** Any process with root on the instance can call the metadata
  endpoint. For process-level isolation, layer container-credential-isolation (EKS IRSA, ECS task
  roles) on top.

## Production hardening

- **Enforce IMDSv2-only** (`HttpTokens: required` in the instance launch template) — blocks the
  SSRF-vulnerable v1 path entirely.
- **Minimal IAM role** — the instance profile should have only the permissions needed for
  Vault/OpenBao auth (typically just `sts:GetCallerIdentity`).
- **Network-level metadata protection** — use `iptables` or security groups to restrict which
  processes can reach 169.254.169.254 (defense in depth against container escapes).
- **Short token TTLs** — configure OpenBao's AWS auth role with short `token_ttl` so a stolen
  Vault token expires quickly.
- **Audit** — OpenBao audit logs show which instance authenticated and what it read.

## Delivery pattern — out of scope

This demo anchors the bootstrap credential (the cloud instance's IAM identity). What *consumes*
the anchored credential — a delivery pattern that materializes or streams the secret to an unchanged
app — is an orthogonal concern addressed by the delivery-pattern family (see
[`../../docs/decision-guide.md`](../../docs/decision-guide.md)). Conflating the two would obscure
this pattern's lesson.

## Related patterns

- **tpm-sealed-bootstrap** (planned) — anchors trust to hardware (TPM silicon) instead of the cloud
  control plane. Use for on-prem / bare-metal where there is no provider-assigned identity.
- **GCP metadata + GCE auth** — same concept, different provider. GCP instances get a service-account
  identity via `metadata.google.internal`.
- **Azure IMDS + managed identity** — same concept. Azure instances call their IMDS for OAuth2 tokens.
- **[dynamic-credential-shim](../../delivery-pattern-demos/dynamic-credential-shim/)** — a delivery
  pattern that can use this bootstrap to authenticate to OpenBao.
- **[cone-of-silence](../../delivery-pattern-demos/cone-of-silence/)** — keeps the delivered secret
  RAM-only (orthogonal to how you bootstrap).

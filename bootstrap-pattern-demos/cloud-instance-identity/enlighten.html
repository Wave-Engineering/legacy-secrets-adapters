Cloud Instance Identity

the machine already has an identity — use it as the bootstrap credential

Every secret-delivery pattern needs a bootstrap credential to authenticate to its key source. On a cloud instance, that credential already exists: the provider assigned an IAM role at launch. No stored secret — the machine's identity IS the credential.

Note: This demo uses a mock that simulates the hypervisor-backed metadata service. In production, the real IMDS is provided by the EC2 hypervisor and is unreachable from outside the instance (link-local 169.254.169.254).

The problem

Every delivery pattern (Cone of Silence, Dynamic Credential Shim, broker-sidecar) bottoms out at the same question: how does the materializer authenticate to its key source without a stored secret? If you store the bootstrap credential in a file, you have just moved the problem one level down. The turtles-to-silicon regress has to end at a root of trust the file-thief cannot reach.

The idea: no stored secret

The cloud provider's control plane is the root of trust. It assigned the instance an IAM role at launch; the metadata endpoint hands out short-lived tokens that prove “I am this instance.” OpenBao's AWS auth method verifies the claim via STS — no shared secret, no file to steal.

How it works: IMDSv2

  EC2 Instance                           169.254.169.254 (hypervisor)
       |                                        |
       |--- PUT /latest/api/token ------------->|   (TTL header required)
       |<-- 200: session-token -----------------|
       |                                        |
       |--- GET /meta-data/iam/...  ----------->|   (X-aws-ec2-metadata-token header)
       |<-- 200: {AccessKeyId, Secret, Token} --|
       |                                        |
  EC2 Instance                           OpenBao / Vault
       |                                        |
       |--- POST /v1/auth/aws/login ----------->|   (signed STS GetCallerIdentity)
       |<-- 200: {client_token, policies} ------|
       |                                        |
       |--- GET /v1/secret/data/... ----------->|   (X-Vault-Token header)
       |<-- 200: {password: ...} ---------------|

Tradeoffs — be honest

Cloud-only

This pattern requires a cloud provider with an instance metadata service and a corresponding Vault/OpenBao auth method. It does not work on bare-metal, on-prem, or in environments without a hypervisor-backed identity.

Trusts the provider

You are trusting the cloud provider's control plane as your root of trust. If the provider is compromised (or coerced), your bootstrap credential is compromised. This is the same trust you already place in the provider for compute, network, and storage.

Metadata service exposure

IMDSv1 was vulnerable to SSRF: any process that could make HTTP requests from the instance could reach 169.254.169.254. IMDSv2 mitigates this (PUT requires a hop-limit header that containers/proxies cannot relay), but misconfiguration can still expose it. Enforce IMDSv2-only and restrict the IAM role's permissions to minimum.

What this does NOT protect against

An attacker with root on the instance can still call the metadata endpoint. This pattern anchors trust to the machine, not to a specific process. For process-level isolation, layer container-credential-isolation (EKS IRSA, ECS task roles) on top.

Related patterns

  • tpm-sealed-bootstrap (planned) — anchors trust to the machine's TPM (silicon). Different root of trust: cloud control plane vs. physical hardware. Use TPM for on-prem/bare-metal; use cloud identity when you have a provider.
  • GCP metadata + GCE auth — same idea, different provider. GCP instances get a service-account identity via metadata.google.internal.
  • Azure IMDS + managed identity — same idea. Azure instances call 169.254.169.254/metadata/identity/oauth2/token.
  • Dynamic Credential Shim — a delivery pattern that can USE this bootstrap to authenticate to OpenBao.
  • Cone of Silence — keeps the delivered secret RAM-only (orthogonal to how you bootstrap).

Delivery pattern — out of scope

This demo anchors the bootstrap credential (cloud instance identity); what consumes the anchored credential — a delivery pattern that materializes the secret to an unchanged app — is the orthogonal delivery-pattern family. See decision-guide.md.

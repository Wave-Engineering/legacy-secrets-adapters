Cloud Instance Identity

the machine already has an identity — use it as the bootstrap credential

legacy-secrets-adapters - a walkthrough

The problem

Every secret-delivery pattern (Cone of Silence, Dynamic Credential Shim) needs a BOOTSTRAP credential to authenticate to its key source. But where does THAT credential come from? If you store it on disk, you've just moved the problem. The turtles-to-silicon regress has to end somewhere.

On a cloud instance, it already has: the machine's identity IS the credential. The cloud provider's control plane assigned it an IAM role when it launched. The metadata endpoint hands out short-lived tokens that prove 'I am this instance'. No stored secret required.

The IMDSv2 flow

AWS Instance Metadata Service v2 uses a two-step flow to prevent SSRF attacks:
Step 1: PUT to get a session token (requires the TTL header).
Step 2: GET with that token to retrieve credentials.

[you@instance cloud-instance-identity]$ curl -s -X PUT -H 'X-aws-ec2-metadata-token-ttl-seconds: 300' http://169.254.169.254/latest/api/token
a4b2c8d9e0f1a2b3c4d5e6f7a8b9c0d1e2f3a4b5c6d7e8f9a0b1c2d3e4f5a6b7
[you@instance cloud-instance-identity]$ curl -s -H 'X-aws-ec2-metadata-token: <token>' http://169.254.169.254/latest/meta-data/iam/security-credentials/demo-instance-role
{
  "Code": "Success",
  "AccessKeyId": "AKIAIOSFODNN7EXAMPLE",
  "SecretAccessKey": "wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY",
  "Token": "FwoGZXIvYXdzE-session-token-0001",
  "Expiration": "2026-01-01T10:00:00Z"
}

The session token and the STS credentials are both short-lived.
No stored secret — just a conversation with the hypervisor.

Authenticate to OpenBao

The materializer uses the AWS credentials to authenticate to OpenBao
via the AWS auth method. OpenBao verifies the identity with STS and
returns a Vault token scoped to the instance's policy.

[you@instance cloud-instance-identity]$ ./materializer.py
[materializer] requesting IMDSv2 session token ...
[materializer] got session token; fetching instance credentials ...
[materializer] got credentials for role (AccessKeyId=AKIAIОСF...)
[materializer] authenticating to OpenBao via AWS auth method ...
[materializer] authenticated (token=hvs.CAES...)
[materializer] reading secret from secret/data/demo/db-password ...
[materializer] got secret; writing to run/secret.json ...
[materializer] done - secret materialized to run/secret.json (mode 0600)

Authenticated and fetched secret — no stored bootstrap credential used.

No secrets on disk

The only file written is the delivery output (the secret itself).
No Vault tokens, no AWS credentials, no session tokens on disk.

[you@instance cloud-instance-identity]$ cat run/secret.json
{
  "password": "S3cr3t-Pg-Pass",
  "username": "app_user"
}
[you@instance cloud-instance-identity]$ grep -r 'dev-only-root-token\|hvs\.' run/ || echo '(none found)'
(none found)

Only the delivery payload is on disk. Auth tokens stayed in memory.

Token rotation is built-in

Each call to the metadata endpoint returns fresh, short-lived credentials.
The cloud control plane handles rotation — no cron job, no lease renewal.

[you@instance cloud-instance-identity]$ # call 1
Token: "FwoGZXIvYXdzE-session-token-0001"
[you@instance cloud-instance-identity]$ # call 2
Token: "FwoGZXIvYXdzE-session-token-0002"

Fresh credentials every time. Rotation is the cloud's native model.

The takeaway

The bootstrap-secret problem — 'what authenticates the authenticator?' — ends at a root of trust the file-thief cannot reach. On a cloud instance, that root is the provider's control plane: the machine's identity IS the credential, assigned at launch, unreachable from outside.

No file to steal, no token to rotate, no secret to seal. The instance metadata endpoint is the trust anchor — backed by the hypervisor, scoped to this machine, short-lived by design.

Delivery pattern - out of scope. This demo anchors the bootstrap credential (cloud instance identity); what consumes it - a delivery pattern that materializes the secret to an unchanged app - is an orthogonal concern (the delivery-pattern family). Why & what: enlighten.html · the catalog.

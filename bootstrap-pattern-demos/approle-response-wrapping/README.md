# AppRole Response-Wrapping

*Deliver the materializer's bootstrap credential as a single-use response-wrapped token — replay-proof, time-limited, no stored secret on disk.*

## Context — when you're here

- A materializer needs to authenticate to its key source (OpenBao/Vault) **without a stored
  secret** on the target machine.
- A trusted deployment pipeline (Ansible, CI/CD, Tower) can deliver a one-time token to the
  target at deploy time.
- No TPM or hardware root of trust is available (or: you want the software-only layer first).

## Forces

- Baking the secret into the image means a stolen image leaks the identity.
- Dropping the secret on disk means a stolen disk leaks it.
- The credential must be **dead on replay** — even if intercepted after delivery.
- The window for interception must be **short** (seconds, not hours).

## Solution

OpenBao's **response-wrapping** generates a SecretID wrapped in a single-use token with a short
TTL. A trusted deployer (Ansible playbook) generates the wrap and delivers the wrapping token to
the target. The materializer unwraps it (consuming the token), obtains the SecretID, authenticates
via AppRole (RoleID + SecretID), and fetches the actual secret. The wrapping token is dead after
first unwrap OR after TTL — replay is impossible.

## How it works

```
  Deployer (Ansible playbook, trusted pipeline)
       │  POST /auth/approle/role/demo-app/secret-id
       │  Header: X-Vault-Wrap-TTL: 30s
       ▼
  Wrapping Token (single-use, 30s TTL)
       │  delivered to target (run/wrapped-token, mode 0400)
       ▼
  Materializer
       │  POST /sys/wrapping/unwrap  →  SecretID
       │  POST /auth/approle/login (RoleID + SecretID)  →  client token
       │  GET  /secret/data/demo-app/config  →  the actual secret
       ▼
  Application (password used in memory, never stored)
```

The RoleID is the non-secret half (safe to store on the target). Only the SecretID crosses the
wire — wrapped, single-use, and time-limited.

./demonstrate.py     # interactive: bring up → wrap+deliver → replay-fail → tear down

## Tradeoffs / residual exposure

- **Requires a trusted deployer.** The entity generating the wrap needs a privileged token
  (it can mint SecretIDs). In production: a CI/CD runner, Ansible Tower, a cloud function with
  an IAM role. The trust has to start *somewhere*.
- **Transit exposure window.** The wrapping token crosses the wire between deployer and target.
  An attacker on the delivery path can race the materializer. Mitigation: short TTL (30s default),
  TLS on the delivery channel, and detection (a failed unwrap by the materializer = someone else
  got there first — alert immediately).
- **No hardware binding.** Response-wrapping proves "the deployer authorized this identity" but
  does NOT prove "this specific machine requested it." For hardware binding, compose with a
  TPM-sealed bootstrap.

## Production hardening

- **Shrink TTL** to the minimum your deployment latency allows (seconds, not minutes).
- **Detect failed unwraps** — if the materializer's unwrap returns 400, someone else consumed
  the token. Wire this to your SIEM/alerting.
- **Scope the deployer's token** with a narrow policy (only `auth/approle/role/*/secret-id`
  + response-wrapping capability).
- **TLS everywhere** — the wrapping token in transit must not be sniffable.
- **Compose with TPM-sealed-bootstrap** to add hardware binding to the software-only wrapping
  guarantee.
- **Short-lived SecretID** — even after unwrap, the SecretID has its own TTL (60s in the demo);
  it must be used quickly.

## Delivery pattern — out of scope

This demo anchors the bootstrap secret (the AppRole SecretID, delivered as a response-wrapped
token). What *consumes* the anchored credential — a delivery pattern that materializes or streams
the secret to an unchanged app (a shim, a sidecar, a Cone) — is an orthogonal concern addressed
by the delivery-pattern family (see
[`../../docs/decision-guide.md`](../../docs/decision-guide.md)). Conflating the two would obscure
this pattern's lesson.

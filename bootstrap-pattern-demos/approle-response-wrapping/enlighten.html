The bootstrap problem

Every delivery pattern (dynamic-credential-shim, cone-of-silence, broker-sidecar) needs a bootstrap credential — how does the materializer authenticate to its own key source? If you bake it into the image, a stolen image leaks the identity. If you drop it on disk, a stolen disk leaks it. You need the credential to be delivered just-in-time, consumed exactly once, and dead after a short window.

How it works

  Deployer (Ansible, CI/CD, Tower)
       │  POST /auth/approle/role/demo-app/secret-id
       │  Header: X-Vault-Wrap-TTL: 30s
       ▼
  Wrapping Token (single-use, 30s TTL)
       │  delivered to target (file, env, pipe)
       ▼
  Materializer
       │  POST /sys/wrapping/unwrap  →  SecretID
       │  POST /auth/approle/login (RoleID + SecretID)  →  client token
       │  GET  /secret/data/demo-app/config  →  the actual secret
       ▼
  Application (password used in memory, never stored)

Why single-use kills replay

Single-use (consumed on unwrap)

The wrapping token is invalidated the moment it is unwrapped. A second unwrap attempt returns HTTP 400. An attacker who intercepts the token after the materializer gets it finds a dead token.

Short TTL (dead after window)

Even if the attacker intercepts the token before the materializer, they have at most 30 seconds to use it. And if the materializer tries next and fails, that is an immediate detection signal — someone else consumed the token.

Detection: failed unwrap = compromise signal

If the materializer's unwrap fails (token already consumed), someone else got there first. This is an active detection mechanism — not just prevention but alerting. In production, wire this to your SIEM.

The transit window

Between generation and consumption, the wrapping token exists in transit. This is the residual exposure: a sufficiently privileged attacker on the delivery channel (the network, the file path) can race the materializer. Shrink the TTL to shrink the window.

Tradeoffs — be honest

Requires a trusted deployer

The entity generating the wrap needs a privileged token (it can mint SecretIDs). In production: a CI/CD runner, Ansible Tower, a cloud function with an IAM role. The trust has to start somewhere.

Transit exposure window

The wrapping token crosses the wire. An attacker on the delivery path can race the materializer. Mitigation: short TTL, TLS, detection (failed unwrap = alert).

No hardware binding

Response-wrapping proves "the deployer authorized this identity" — it does NOT prove "this specific machine requested it." For hardware binding, compose with a TPM-sealed bootstrap (the materializer presents a TPM attestation before the deployer wraps).

Related concepts

  • dynamic-credential-shim — a delivery pattern that consumes the AppRole token this bootstrap provides (orthogonal axis).
  • tpm-sealed-bootstrap (planned) — hardware-binds the identity so even the deployer can't impersonate the target without its TPM.
  • cone-of-silence — keeps the fetched secret RAM-only once authenticated.

Delivery pattern — out of scope

This demo anchors the bootstrap secret (the AppRole SecretID, delivered as a response-wrapped token). What consumes the anchored credential — a shim that materializes or streams the secret to an unchanged app — is an orthogonal concern addressed by the delivery-pattern family. See decision-guide.md.

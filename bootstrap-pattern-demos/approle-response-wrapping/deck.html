AppRole Response-Wrapping

a single-use wrapped token bootstrap — replay-proof, time-limited, no stored secret

legacy-secrets-adapters · a walkthrough

The bootstrap problem

A materializer needs to authenticate to its key source (OpenBao) without a stored secret. If you bake the secret into the image or drop it on disk, anyone who steals the disk steals the identity. The question: how do you deliver a credential that is DEAD on replay and DEAD after a short window?

Answer: response-wrap a SecretID. The wrapping token is single-use (consumed on first unwrap) and short-lived (TTL). A trusted deployer generates it; the materializer consumes it exactly once.

Bring up OpenBao + configure AppRole

[deployer] Real OpenBao in dev mode. The root token here is the obvious dev value
this pattern replaces in production.

Deployer wraps SecretID (Ansible)

[deployer] The Ansible playbook generates a response-wrapped SecretID
and delivers the wrapping token to the target. TTL=30s, single-use.

[you@host approle-response-wrapping]$ ansible-playbook -i inventory.yml playbook.yml

TASK [Ensure run directory exists] ***
ok: [localhost]

TASK [Generate response-wrapped SecretID] ***
ok: [localhost]

TASK [Deliver wrapped token to target] ***
changed: [localhost]

TASK [Report delivery] ***
ok: [localhost] => msg: "Wrapped SecretID delivered to run/wrapped-token (TTL=30s, single-use, accessor=hmac-v1:abc...)"

Wrapping token delivered. It can be unwrapped exactly once, within 30s.

Materializer unwraps + authenticates

[target] The materializer reads the wrapping token, unwraps it to get the
SecretID, then authenticates via AppRole (RoleID + SecretID) and fetches the secret.

[you@host approle-response-wrapping]$ python3 materializer.py
[materializer] unwrapping token to obtain SecretID ...
[materializer] SecretID obtained (single-use token consumed)
[materializer] authenticating via AppRole (RoleID + SecretID) ...
[materializer] authenticated — got client token
[materializer] fetching secret from secret/data/demo-app/config ...
[materializer] secret retrieved: keys=['password', 'host']
[materializer] (password used in memory — never printed, never logged)

Materializer authenticated without a stored secret.

Replay fails — single-use

[attacker] What if someone steals the wrapping token after delivery?
They try to unwrap it again:

[you@host approle-response-wrapping]$ # attempt to unwrap the same token a second time
Error: wrapping token is not valid or does not exist (HTTP 400)

Single-use: consumed on first unwrap. Replay is impossible.

Expired token fails — TTL

[attacker] What if they intercept the token before the materializer gets it,
but wait too long?

[you@host approle-response-wrapping]$ # generate a token with TTL=2s, wait 3s, attempt unwrap
Error: wrapping token is not valid or does not exist (HTTP 400)

TTL expired: the token is dead. The window to intercept is tiny.

The takeaway

A response-wrapped SecretID is single-use (consumed on unwrap) and short-lived (TTL). Together they make the bootstrap credential replay-proof: an attacker who intercepts the token has a narrow window to use it, and if the materializer gets there first the token is already gone.

The deployer needs a privileged token to generate wraps — this is the trust anchor. In production that's a CI/CD pipeline with limited scope, an Ansible Tower credential, or a cloud identity. The RoleID is the non-secret half; only the wrapped SecretID crosses the wire.

Delivery pattern — out of scope. This demo anchors the bootstrap; what consumes the anchored credential (a shim, a sidecar, a Cone) is the delivery-pattern family. Why & what: enlighten.html · the catalog.

The Cone of Silence

a legacy-secrets adapter pattern  ·  “a low-effort, RAM-only zone your secret never leaves”

The secret is never stored in the clear — it is only ever briefly shown.

The problem

A legacy application reads its credentials from a plaintext file on disk — say /etc/secrets.json. Anyone who can read that disk — another process, a backup, a container image layer, a stray git add, a stolen drive — gets the password. And the fix can't change the application: it is not going to be re-released.

The idea, side by side

Keep the app's file at the path it always read. Just change what is behind that path.

On disk

permanentcopyable Only ciphertext lives here. To a thief it's gibberish.

In RAM (tmpfs)

momentarygone on power-off The plaintext appears only here, only while the app needs it.

Appendix — Transparency in Secrecy

RAM is not a vault. A plaintext file in tmpfs is still readable by root, by anything that can ptrace the reader or peek at /proc/<pid>/mem, by processes that share its mount namespace — and it can leak to swap or a core dump. Being honest about that is the point. Each avenue has a mitigation, and they layer:

Temporal mounting

Populate the Cone only for the brief window the app actually needs the secret — decrypt in, let it read, wipe out — so the plaintext exists for seconds, not for the process's whole life.

  • Smallest possible exposure window
  • Most of the time there is simply nothing to read
  • Must orchestrate the timing precisely
  • Breaks if the app re-reads or holds the file open

Dedicated user

Run the reader as its own service account; the Cone file is mode 0400 owned by that uid, so other users and other services simply cannot read it.

  • Simple, standard, least-privilege
  • Stops every non-root neighbor cold
  • root can still read it
  • Doesn't stop privileged ptrace; needs the account provisioned

systemd sandboxing

Lock the service down declaratively: ProtectSystem=strict, ProtectHome, PrivateTmp, NoNewPrivileges, ProtectProc=invisible, InaccessiblePaths=, RestrictAddressFamilies=. Even a compromised neighbor service can't reach the secret or the process.

  • Defense-in-depth, no code changes
  • Declarative; shrinks the blast radius
  • Over-tightening can break the app
  • Doesn't constrain root or the kernel

Mount namespaces / unshare

Put the Cone in a private mount namespace (PrivateMounts=yes / PrivateTmp=yes, or unshare -m) so the mount — and the plaintext on it — is literally invisible to processes outside the namespace.

  • The secret mount doesn't exist in anyone else's filesystem view
  • Strong isolation between services
  • Host root can still nsenter in
  • More moving parts; harder to debug

Beyond isolating the mount, close the subtler paths the plaintext can take once it's in RAM — and the one thing the cards above punt on, root itself:

No swap / encrypted swap / mlock

Under memory pressure the kernel can page tmpfs out to the swap device — landing your plaintext on a disk that does persist. Close it: disable swap, encrypt it (LUKS or zram), or mlock the secret pages so they can't be evicted.

  • Removes the "plaintext silently written to swap" leak
  • Encrypted swap protects all RAM spillover, not just this secret
  • mlock needs CAP_IPC_LOCK / a raised RLIMIT_MEMLOCK
  • Disabling swap risks OOM kills; encrypted swap adds setup

Restrict ptrace

A process running as the same user (or root) can attach to the reader and read the decrypted secret straight out of its memory via /proc/<pid>/mem. Set kernel.yama.ptrace_scope=2 (or 3) to forbid it, and mount /proc with hidepid=2 so other users can't even enumerate the process.

  • Closes the live-memory read path between same-user processes
  • hidepid also hides cmdlines/env that may carry secrets
  • Breaks debuggers/profilers (gdb, strace) in dev
  • root can re-raise the scope — not a defense against root

Disable core dumps

If the reader crashes, a core dump can capture the decrypted secret to disk. Turn dumps off for the service: LimitCORE=0 (systemd), prctl(PR_SET_DUMPABLE, 0), and a sane kernel.core_pattern — so a crash can't spill the secret.

  • Removes the "secret ends up in a crash file" leak
  • Cheap and declarative in a systemd unit
  • Loses post-mortem debugging for that service
  • Doesn't stop a live read — pair it with the ptrace card

SELinux / MAC confinement constrains even root

Discretionary permissions (0400 + a uid) don't constrain root. Mandatory Access Control does: label the Cone and the secret with a dedicated SELinux type and write policy so only the reader's domain may read it — a root process in the wrong domain is denied. On RHEL this is already enforcing; you're extending policy you run anyway.

  • Constrains even root — the answer to the caveat the other cards punt on
  • Native and enforcing on RHEL; no new runtime dependency
  • Writing/maintaining policy is real work; a mislabel denies legitimately
  • A process running in the reader's domain is still trusted

But what about writes?

The Cone as shown is read-only — the app reads its config. Some apps also write: runtime state, or rotating their own credential. What to do depends on two questions: do the writes need to persist, and are they secret?

  • Disposable runtime state → let it live and die in the Cone. Nothing to persist, nothing to encrypt. Simplest path: do nothing.
  • Perma-mount + broker process → keep the Cone mounted for the app's lifetime, owned by a long-running broker that seeds it from the ciphertext and re-encrypts writes back to disk. The app reads and writes freely; only the broker touches the at-rest store. Persists writes, app unchanged — at the cost of broker complexity and a longer plaintext window.
  • inotify write-back → watch the file; on close/modify, re-encrypt to the at-rest ciphertext. Lighter than a full broker — mind write races and partial writes.
  • FUSE encrypting filesystem (gocryptfs / EncFS) → encrypt-on-write, decrypt-on-read, transparently; files at rest are always ciphertext. The general read-write answer with no app change — at the cost of a FUSE dependency and some performance.
  • App rotates its own secret → the broker captures the write and pushes the new value to OpenBao, keeping the vault the source of truth. Supports self-rotating apps; needs bidirectional sync and conflict handling.

In production

The Cone is a per-service tmpfs at /run/<service> (systemd RuntimeDirectory — the directory locked to 0700 via RuntimeDirectoryMode, the secret file written mode 0400, owned by the app, auto-removed on stop). The decryption key never touches the disk — it is sealed to the machine's TPM, or handed over by OpenBao (the same vault already issuing our SSH certificates), which can also rotate the secret behind the scenes.

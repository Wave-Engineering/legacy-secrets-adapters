Dynamic Credential Shim

a static DB password → an OpenBao-managed credential that rotates, so a leak self-expires

legacy-secrets-adapters · a walkthrough

The problem

A legacy app reads a STATIC Postgres password from a file and won't be re-released. That password is valid forever — so a single leak (a backup, a log, a stolen disk) is valid forever too. We want the credential to ROTATE and a leak to self-expire — without touching the app.

Bring up OpenBao + Postgres

Real OpenBao (dev mode) + Postgres in containers. The OpenBao token and the PG superuser
password here are obvious dev values — bootstrap secrets, out of scope (see the README).

[you@host dynamic-credential-shim]$ docker compose up -d
 ✔ Container dynamic-credential-shim-postgres-1  Healthy
 ✔ Container dynamic-credential-shim-openbao-1   Started

Configure the database secrets engine and a STATIC role — OpenBao now owns and rotates
the password of one fixed Postgres user, app_pg_user:

[you@host dynamic-credential-shim]$ ./setup.sh
→ creating the fixed app role OpenBao will manage (app_pg_user)
→ enabling + configuring the database secrets engine
→ rotating OpenBao's own admin password (now no human knows it)
→ defining the STATIC role (OpenBao owns + rotates app_pg_user's password)
✅ setup complete

The managed credential works

The shim fetches the current managed credential and writes the file the reader reads:

[you@host dynamic-credential-shim]$ ./shim.py
→ shim: wrote OpenBao-managed credential for 'app_pg_user' to run/secrets.json

The unchanged legacy reader connects — it has no idea the password is managed or rotates:

[you@host dynamic-credential-shim]$ ./legacy_reader.py
[legacy-reader] opened secrets.json; connecting to 127.0.0.1:55432/appdb as app_pg_user ...
[legacy-reader] connected OK as app_pg_user|2026-01-01 09:00:00+00
[legacy-reader] (password used in memory to connect — never printed, never logged)

Rotate — and the leaked copy dies

Say an attacker exfiltrates a copy of the current password. Then OpenBao rotates the
static role (here we force it; in production it's a schedule):

[you@host dynamic-credential-shim]$ bao write -f database/rotate-role/app-static
Success! Data written to: database/rotate-role/app-static

The exfiltrated copy no longer authenticates:

[you@host dynamic-credential-shim]$ psql 'postgresql://app_pg_user:<exfiltrated>@127.0.0.1:55432/appdb' -tAc 'select 1'
psql: error: connection to server at "127.0.0.1", port 55432 failed:
FATAL:  password authentication failed for user "app_pg_user"

The leak self-expired. A stolen credential is dead within one rotation window.

The app keeps working — untouched

The shim re-materializes the new password; the reader (unchanged) reconnects:

[you@host dynamic-credential-shim]$ ./shim.py
→ shim: wrote OpenBao-managed credential for 'app_pg_user' to run/secrets.json
[you@host dynamic-credential-shim]$ ./legacy_reader.py
[legacy-reader] opened secrets.json; connecting to 127.0.0.1:55432/appdb as app_pg_user ...
[legacy-reader] connected OK as app_pg_user|2026-01-01 09:00:00+00
[legacy-reader] (password used in memory to connect — never printed, never logged)

The takeaway

Static password → a leak is valid forever. An OpenBao-managed STATIC role → a fixed username (pool-friendly), a password OpenBao rotates, and a leaked copy that self-expires within a rotation window. The app never changed — only what's behind its file did.

Dynamic roles (a brand-new Postgres user per short lease) are a small extension of this — left to the reader. They harden the leak window further but the changing username can break a legacy connection pool; see NOTES.md.

Bootstrap secret — out of scope. The OpenBao token here is an obvious dev value; how the shim would really authenticate is an orthogonal concern (the bootstrap-secret pattern family). Why & what: enlighten.html · deep-dive: NOTES.md · the catalog.
